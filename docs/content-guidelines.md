# 内容表达规范

本文件记录 Percy Site 前台内容的表达方向，用于保持首页、关于页、文章和后台内容字段的长期一致性。

## 总体原则

- 表达要克制、清晰、具体，避免过度包装。
- 先讲真实经历和判断，再讲技术名词。
- 技术能力应服务于产品体验、工程质量和问题解决，不把工具清单当作核心卖点。
- 文案要像个人站点，而不是简历模板或营销落地页。
- 中英文混排时，英文短语用于必要概念，不为了显得技术化而堆叠。

## 首页

首页承担第一印象，重点回答：

```txt
我是谁
我关注什么
我如何做产品和工程
```

推荐方向：

- 身份行可以直接表达职业与当前方向，例如 `Frontend Engineer / AI Product Builder`。
- 主描述应强调信息结构、交互状态、视觉细节和工程实现之间的关系。
- 标签用于补充工作方式和审美，不重复身份行。

当前标签方向：

```txt
Minimal Design
Clear Interfaces
Reliable Systems
Product Thinking
```

避免：

- 把技术栈作为首页主角。
- 使用过于抽象的词，例如只有 `Clean`、`Typed`、`System`，但不说明价值。
- 使用过度营销化表达，例如“赋能”“领先”“极致”等。

## 关于页

关于页可以比首页更具体，但仍然保持自然口吻。

推荐结构：

1. 工作经历和经验来源。
2. 技术观、产品观和 AI 应用开发方向。
3. 工作之外的生活状态。

写工作经历时，避免只给结论。应补出经历和判断之间的链路，例如：

```txt
复杂 B 端业务系统
流程、权限、状态和协作问题
用户体验、产品落地和工程质量之间的平衡
```

写 AI 应用开发时，重点不只是“使用 AI”，而是：

- 模型能力如何转化为产品体验。
- 产品交互如何承接不确定输出。
- 工程实现如何让 AI 应用可用、稳定、可维护。

生活兴趣可以独立成段，用来补充个人状态，不需要写成爱好清单。

## 文章内容

文章应优先提供真实思考和可复用经验。

推荐写法：

- 先说明问题背景。
- 再讲判断和取舍。
- 最后给出可执行的实践方式。

适合的文章方向：

- 前端工程实践。
- 组件设计和界面状态。
- AI 应用开发。
- 产品体验与工程落地。
- 个人项目复盘。

避免：

- 只罗列 API 和配置。
- 为了追热点而写缺少实践背景的内容。
- 标题夸张但正文没有足够信息密度。

## 后台内容字段

后台编辑文章、资源和描述信息时，应保持字段语义清楚。

- 标题：直接表达主题，不做标题党。
- 摘要：说明文章解决什么问题或提供什么判断。
- 分类：用稳定的大类，不频繁制造新分类。
- 标签：补充具体主题，数量保持克制。
- 图片描述：说明图片内容或上下文，避免无意义文件名。

## 语言风格

推荐：

```txt
清晰
克制
具体
有判断
```

避免：

```txt
空泛
堆词
自夸
过度技术化
过度营销化
```
